# inVision U — README

## О проекте

**inVision U** — это портал отбора абитуриентов с AI-поддержкой для первичного анализа заявок.

Система решает две задачи:

1. **Кандидатский сценарий** — сбор заявки по шагам: личные данные, бэкграунд, эссе, интервью и согласие.
2. **Сценарий комиссии** — просмотр заявок, AI-анализ, evidence-блоки, флаги, сравнение двух кандидатов и обзор аналитики.

Проект сделан на **Next.js 14 + TypeScript**, хранит данные в **Supabase**, а для извлечения сигналов из текста использует **OpenAI API**.

---

## Что умеет система

### Для кандидата
- многошаговая подача заявки;
- ввод личных данных, GPA, активностей и достижений;
- написание эссе;
- ответы на интервью-вопросы;
- запись голосового ответа через браузерный speech recognition;
- защита от дублей по email;
- подтверждение согласия на обработку данных и AI-анализ.

### Для комиссии
- вход по паролю комиссии;
- список заявок со статусами;
- просмотр детальной карточки кандидата;
- AI-оценка по критериям:
  - мотивация;
  - лидерство;
  - опыт;
  - рост;
- флаги, риски, противоречия и сильные стороны;
- evidence-блоки с цитатами и объяснением, почему они влияют на оценку;
- compare view для двух кандидатов;
- сводная аналитика по заявкам.

## Как устроен проект

### 1. Подача заявки
Кандидат проходит последовательные шаги в `/candidate/*`:
- `personal` — персональные данные;
- `background` — активности и достижения;
- `essay` — эссе;
- `interview` — интервью;
- `consent` — согласие.

На финальном шаге данные отправляются в `POST /api/candidates`.

### 2. Защита от дублей
Перед созданием новой записи сервер:
- нормализует email;
- ищет последнюю заявку с таким email;
- если заявка уже существует, возвращает `409` и не создаёт дубль.

### 3. AI-пайплайн оценки
После сохранения заявки запускается пайплайн `runPipeline()`:
- **baseline** — детерминированная стартовая оценка;
- **llm-extractor** — извлечение сигналов из эссе и интервью через OpenAI;
- **aggregator** — сбор итогового скоринга по правилам проекта.

Результат сохраняется в поле `scoring`, а статус заявки меняется на `scored`.

### 4. Панель комиссии
После авторизации комиссия получает доступ к:
- списку заявок;
- детальной карточке кандидата;
- сравнению двух кандидатов;
- аналитике.

## Переменные окружения

Создайте файл `.env.local` в корне проекта:

```env
NEXT_PUBLIC_SUPABASE_URL=your_supabase_project_url
SUPABASE_SERVICE_ROLE_KEY=your_supabase_service_role_key
OPENAI_API_KEY=your_openai_api_key
COMMITTEE_PASSWORD=your_committee_password
```

### Что означает каждая переменная
- `NEXT_PUBLIC_SUPABASE_URL` — URL вашего Supabase-проекта.
- `SUPABASE_SERVICE_ROLE_KEY` — серверный ключ Supabase для API-роутов.
- `OPENAI_API_KEY` — ключ OpenAI для AI-анализа и транскрибации.
- `COMMITTEE_PASSWORD` — пароль входа для комиссии.

> Важно: `SUPABASE_SERVICE_ROLE_KEY` нельзя публиковать на клиенте. В проекте он используется только на сервере.

## API-роуты

### `POST /api/candidates`
Создаёт новую заявку.

Что делает:
- валидирует поля;
- проверяет дубль по email;
- сохраняет заявку;
- запускает AI-пайплайн;
- возвращает `candidate_id`.

### `GET /api/committee`
Возвращает список заявок для комиссии.

Требует заголовок:

```http
x-committee-password: <COMMITTEE_PASSWORD>
```

### `GET /api/committee/:id`
Возвращает полную карточку кандидата.

### `DELETE /api/committee/:id`
Удаляет заявку.

### `GET /api/committee/compare?ids=id1,id2`
Возвращает две заявки для compare view.

### `POST /api/transcribe`
Транскрибирует аудио через OpenAI.

## Как работает AI-оценка

AI-пайплайн собирает:
- оценки по критериям;
- confidence;
- flags;
- contradictions;
- strengths;
- concerns;
- evidence;
- краткие reasoning-блоки;
- review по эссе и интервью;
- assessment на предмет естественности текста.

Итоговая логика строится так:
1. `baseline.ts` задаёт начальную оценку на основе формальных сигналов;
2. `llm-extractor.ts` извлекает качественные сигналы из текста;
3. `aggregator.ts` собирает финальный балл `overall` и субоценки.

### Evidence
Для интерфейса комиссии evidence проходит дополнительную подготовку в `lib/evidence-display.ts`:
- очищаются плохие или нерелевантные цитаты;
- отбрасываются вопросы и prompt-фразы;
- при необходимости цитаты восстанавливаются из сырых текстов кандидата;
- удаляются дубли.

Это нужно, чтобы на экране комиссии показывались фрагменты, максимально похожие на реальные слова кандидата.

## Безопасность

В проекте уже предусмотрены базовые меры:
- `service_role` используется только в серверных роутингах;
- для доступа комиссии нужен пароль;
- в Supabase включён RLS;
- анонимному пользователю разрешён только insert заявки с `consent = true`.

Что стоит усилить перед продом:
- заменить простой пароль комиссии на полноценную аутентификацию;
- добавить rate limit на API;
- логировать действия комиссии в `audit_log`;
- вынести пайплайн в очередь/background worker;
- ограничить размер и частоту аудио-запросов.

## Частые проблемы

### 1. Не загружается список комиссии
Проверьте:
- заполнен ли `COMMITTEE_PASSWORD`;
- совпадает ли пароль в интерфейсе и на сервере;
- доступен ли Supabase.

### 2. Заявка не создаётся
Проверьте:
- заполнено ли согласие;
- не существует ли уже заявка с этим email;
- корректен ли `NEXT_PUBLIC_SUPABASE_URL`;
- корректен ли `SUPABASE_SERVICE_ROLE_KEY`.

### 3. AI-оценка не появляется
Проверьте:
- есть ли `OPENAI_API_KEY`;
- не упал ли вызов OpenAI в логах;
- не сохранился ли статус заявки как `error`.

### 4. Голосовой ввод не работает
Компонент `VoiceRecorder` использует браузерный speech recognition.
Обычно нужен Chrome или Safari и доступ к микрофону.
